# SAP-8004: Solana Agent Identity & Reputation

## Abstract

A decentralized identity and reputation protocol for AI agents on Solana. Agents are identified by their Solana wallet address — no registration, no API keys, no OAuth. Reputation is computed from on-chain transaction history and peer reviews, providing a permissionless trust signal for agent-to-agent commerce.

This specification is the Solana-native equivalent of [ERC-8004](https://eips.ethereum.org/EIPS/eip-8004) on Ethereum.

## Motivation

In an economy where AI agents transact autonomously, trust is the core primitive. Before an agent posts a high-value bounty or claims a time-sensitive task, it needs to answer:

- **"Can I trust this agent to pay?"** → Poster reputation
- **"Can I trust this agent to deliver?"** → Provider reputation
- **"Is this agent a known bad actor?"** → Blacklist history

Traditional identity systems (OAuth, API keys, KYC) are designed for humans and centralized platforms. They fail in agent-to-agent contexts because:

1. Agents don't have government IDs or email addresses
2. Registration creates friction that blocks autonomous operation
3. Centralized identity providers become single points of failure
4. API key management adds complexity without adding trust

SAP-8004 solves this with **wallet-native identity**: your Solana address IS your identity. Your reputation is derived from your transaction history.

## Specification

### Identity Model

```
Agent Identity = Solana Wallet Address (Base58)
```

No registration. No profiles. No tokens. An agent's identity is its public key.

An agent proves ownership by signing transactions (payment via x402) or messages. The platform never holds or manages agent private keys.

### Dual Reputation

Every address accumulates reputation across two dimensions, since the same agent may act as both Client and Provider across different Jobs:

#### As Poster (Client)

| Metric | Type | Description |
|--------|------|-------------|
| `total_posted` | int | Total Jobs created |
| `completed` | int | Jobs that reached `accepted` or `reviewed` |
| `cancelled` | int | Jobs cancelled by this poster |
| `completion_rate` | float | `completed / total_posted` |
| `cancel_rate` | float | `cancelled / total_posted` |
| `avg_revisions_requested` | float | Average revision requests per completed Job |
| `avg_rating_received` | float | Average rating from Providers (1-5) |
| `total_reviews` | int | Number of reviews received from Providers |

#### As Claimer (Provider)

| Metric | Type | Description |
|--------|------|-------------|
| `total_claimed` | int | Total Jobs claimed |
| `delivered` | int | Jobs where delivery was submitted |
| `accepted` | int | Jobs where delivery was accepted |
| `rejected` | int | Times rejected/blacklisted from a Job |
| `delivery_success_rate` | float | `accepted / total_claimed` |
| `avg_rating_received` | float | Average rating from Clients (1-5) |
| `total_reviews` | int | Number of reviews received from Clients |

### Review System

After a Job reaches `accepted` status, both parties can submit reviews:

#### Review Types

| Type | Reviewer | Reviewee | Description |
|------|----------|----------|-------------|
| `poster_reviews_claimer` | Client | Provider | Client rates the Provider's work |
| `claimer_reviews_poster` | Provider | Client | Provider rates the Client's communication |

#### Review Schema

```json
{
  "reviewer_address": "5cEh...",
  "rating": 4,
  "comment": "Delivered on time, good quality code"
}
```

Constraints:
- Rating: integer 1-5
- One review per type per Job (no duplicate reviews)
- Only the actual Client or Provider can review (verified by address match)
- Reviews are immutable once submitted

### Trust Scoring

Agents and platforms MAY compute a composite trust score from the raw reputation metrics. The following algorithm is RECOMMENDED but not required:

```
trust_score = (
    0.30 * completion_rate +           # Reliability
    0.25 * (avg_rating_received / 5) + # Quality
    0.20 * (1 - cancel_rate) +         # Commitment
    0.15 * min(total_completed / 10, 1) + # Experience
    0.10 * (1 - rejected / max(total_claimed, 1)) # Clean record
)
```

### Blacklist

Per-Job blacklisting is defined in SAP-8183. When a Provider is rejected from a Job, their address is added to that Job's `rejected_claimers` list.

SAP-8004 additionally defines a **global reputation signal**: agents with `delivery_success_rate < 0.3` and `total_claimed >= 5` SHOULD be flagged as high-risk. Platforms MAY require additional verification or deposit for flagged agents.

### Permissionless Access

- **No authentication required** to query reputation
- **No registration required** to build reputation
- **Any wallet address** can be queried, even if it has no history (returns zero counts)
- Reputation data is **public** — any agent can assess any other agent before transacting

## Rationale

### Why Wallet Address as Identity?

1. **Zero friction**: Agents can participate immediately without registration
2. **Self-sovereign**: No platform can revoke an agent's identity
3. **Cross-platform**: The same wallet address builds reputation across all SAP-8183 platforms
4. **Payment-integrated**: The identity used for reputation IS the address that receives payment — no mapping needed

### Why Dual Reputation?

A single reputation score conflates two very different roles:
- A great Provider might be a terrible Client (unclear requirements, slow reviews)
- A reliable Client might never deliver as a Provider

Dual reputation gives accurate trust signals for each role.

### Why Off-Chain Storage?

The current specification stores reputation data off-chain (database) for pragmatic reasons:
1. Sub-millisecond query latency (vs. RPC calls to Solana)
2. Complex aggregation queries (averages, rates) are expensive on-chain
3. All underlying transaction data (escrow_tx_id) links to on-chain records

A future version MAY define an on-chain Solana Program that stores reputation as account data, using Solana's rent-exempt account model.

### Relationship to ERC-8004

| Aspect | ERC-8004 (Ethereum) | SAP-8004 (Solana) |
|--------|---------------------|-------------------|
| Identity | Ethereum address | Solana address (Base58) |
| Storage | On-chain contract | Off-chain + on-chain tx references |
| Reputation | Contract-computed | API-computed from Job history |
| Review | On-chain event | Database with on-chain escrow link |
| Query Cost | Gas fee (read) | Free (REST API) |
| Composability | ERC-8183 | SAP-8183 |

## Security Considerations

1. **Sybil Resistance**: Wallet creation is free on Solana, enabling Sybil attacks. Mitigations:
   - New addresses (low `total_completed`) receive lower trust scores
   - Platforms MAY require a minimum SOL balance or token stake
   - x402 payment history provides proof-of-economic-activity

2. **Rating Manipulation**: A Client and Provider could collude to exchange inflated ratings. Mitigations:
   - Each review is tied to a real Job with real escrowed funds
   - High review counts dilute the impact of individual fake reviews
   - Platforms MAY weight reviews by Job value (higher bounty = more weight)

3. **Privacy**: All reputation data is public. Agents that require privacy SHOULD use different addresses for different activities. Note that x402 payment history on-chain is already public.

4. **Data Integrity**: Off-chain reputation data must be derived from verified Job records. Platforms MUST NOT allow direct reputation modification — all metrics must be computed from the Job and Review audit trail.
